# 内容审核、隐私与安全

## 1. 设计目标

Gift 涉及亲密关系、照片、聊天记录、纪念日、地址、订单等敏感信息，因此审核、隐私和安全必须从第一版开始设计。

目标：

- 防止平台变成广告站。
- 防止社区变成炫富或攻击场。
- 降低用户发布时的隐私泄露风险。
- 让用户知道内容如何被审核和处理。
- 保护普通用户、内容作者和被送礼对象。

## 2. 内容质量标准

高质量礼物内容应该满足：

- 有真实照片。
- 标题具体，能体现关系和场景。
- 场景清楚。
- 预算明确或给出区间。
- 说明为什么送。
- 有对方反应。
- 有准备过程。
- 有适合人群。
- 有不适合情况或避坑提醒。
- 不是纯广告。
- 不是简单商品搬运。

低质量内容：

- 只有商品名和购买链接。
- 只有一句“很好用”。
- 没有任何送礼背景。
- 图片明显来自商品详情页。
- 标题党。
- 内容重复。

违规内容：

- 隐私泄露。
- 广告营销。
- 盗图。
- 虚假内容。
- 恶意攻击。
- 低俗内容。
- 违法危险行为。

## 3. 审核对象

MVP 需要审核：

- 礼物故事正文。
- 礼物图片。
- 评论。
- 举报内容。
- 用户头像和昵称，基础检查。

P1 增加：

- 礼物改造版本。
- 感谢内容。
- 礼物计划分享内容，如果支持公开。

## 4. 内容状态

```text
draft 草稿
pending 待审核
approved 已通过
rejected 已驳回
hidden 已隐藏
deleted 已删除
```

状态流转：

```text
draft → pending → approved
                 → rejected
approved → hidden
approved → deleted
hidden → approved
```

## 5. 审核维度

审核员检查：

| 维度 | 检查问题 | 处理建议 |
| --- | --- | --- |
| 隐私 | 是否有姓名、地址、电话、订单号、聊天记录 | 驳回或要求打码 |
| 人脸 | 是否展示对方人脸且未说明同意 | 提醒或驳回 |
| 广告 | 是否堆商品链接、商家话术明显 | 驳回或隐藏 |
| 盗图 | 是否明显使用商品图或搬运图 | 驳回 |
| 虚假 | 是否夸大、伪装测评 | 驳回 |
| 攻击 | 是否嘲讽、辱骂、歧视 | 驳回或隐藏 |
| 低俗 | 是否包含低俗色情暗示 | 驳回 |
| 违法 | 是否涉及违法危险行为 | 删除并限制账号 |
| 质量 | 是否缺少核心字段 | 驳回补充 |

## 6. 发布前隐私提醒

发布前必须展示提醒：

```text
请确认图片和文字中没有包含真实姓名、地址、电话、订单号、聊天记录等隐私信息。
如果礼物涉及对方照片，建议获得对方同意或进行模糊处理。
```

用户需要勾选：

```text
我已确认内容不包含未处理的隐私信息。
```

## 7. 隐私保护功能

MVP 必须支持：

- 匿名发布。
- 隐藏具体价格。
- 隐藏具体日期。
- 图片隐私提醒。
- 内容举报。
- 作者删除或撤回内容。

P1 建议支持：

- 图片局部涂抹。
- 人脸模糊。
- 敏感文字涂抹。
- 发布前隐私检查清单。

P2 可支持：

- 图片自动识别人脸、地址、电话、订单号。
- OCR 识别聊天记录和敏感文字。
- AI 隐私风险提示。

## 8. 反广告机制

### 8.1 风险表现

- 大量商品链接。
- 商家伪装普通用户。
- 评论区全是求链接。
- 内容像测评软文。
- 批量发布相似内容。
- 图片来自商品详情页却伪装成实拍。

### 8.2 MVP 策略

- 新用户限制外链。
- 正文中链接数量限制。
- 评论中禁止或限制外链。
- 购买来源不是核心展示字段。
- 审核后台标记“疑似广告”。
- 用户可举报广告营销。
- 首页精选不选择明显导购内容。

### 8.3 可疑信号

- 同一用户短时间发布多篇相似内容。
- 多篇内容出现同一店铺或短链。
- 标题和正文过度强调价格优惠。
- 图片质量像商品详情图。
- 评论区集中出现购买引导。

## 9. 反炫富机制

风险表现：

- 高价礼物获得更多曝光。
- 用户开始攀比。
- 低预算用户不敢分享。

产品策略：

- 首页固定展示“低预算高心意”。
- 不按价格做默认排序。
- 不突出“最贵”。
- 点赞命名为“很用心”。
- 精选内容优先考虑情绪价值。
- 预算用区间展示。
- 支持隐藏具体价格。

排序策略：

- 低预算优质内容有稳定曝光池。
- 热度排序使用多样性打散。
- 高价礼物不因预算高获得额外权重。

## 10. 评论区安全

评论规则：

- 不允许辱骂作者或被送礼对象。
- 不允许索要隐私信息。
- 不允许恶意嘲讽低预算礼物。
- 不允许广告链接。
- 不允许人肉搜索。

系统限制：

- 评论长度 2-1000 字。
- 新用户评论频率限制。
- 同一内容重复提交限制。
- 敏感词基础检查。
- 评论支持举报和隐藏。

## 11. 举报处理

举报类型：

```text
侵犯隐私
广告营销
虚假内容
恶意攻击
低俗内容
盗图
重复内容
其他
```

处理动作：

- 不处理，标记无效。
- 隐藏内容。
- 驳回并要求作者修改。
- 删除内容。
- 限制用户发布。
- 封禁账号。

处理 SLA：

| 类型 | 目标处理时间 |
| --- | --- |
| 隐私泄露 | 4 小时内优先处理 |
| 违法危险 | 4 小时内优先处理 |
| 广告营销 | 24 小时内 |
| 盗图 | 24 小时内 |
| 普通争议 | 48 小时内 |

## 12. 用户安全与账号安全

账号策略：

- 密码使用成熟哈希算法。
- 登录失败次数限制。
- 管理员账号建议开启二次验证。
- 管理接口必须校验角色。
- 高风险操作记录审计日志。

高风险操作：

- 审核通过 / 驳回。
- 隐藏内容。
- 删除内容。
- 修改标签。
- 合并标签。
- 封禁用户。
- 修改精选内容。

## 13. Web 安全

必须防护：

- XSS
- CSRF
- SQL 注入
- 文件上传攻击
- 越权访问
- 暴力登录
- API 滥用

实践：

- 所有用户输入转义输出。
- 富文本如有需要必须使用白名单清洗。
- 数据库查询通过 ORM 或参数化。
- 写接口校验 CSRF 或使用安全 SameSite Cookie。
- 上传文件限制 MIME、扩展名、大小。
- 禁止上传可执行文件和 SVG。
- 对象存储 key 不使用用户原始文件名。
- API 限流。
- 管理后台单独权限校验。

## 14. 数据隐私

### 14.1 数据最小化

只收集必要数据：

- 账号信息。
- 用户公开资料。
- 用户主动发布的内容。
- 基础行为事件。

不应收集：

- 不必要的真实姓名。
- 不必要的身份证明。
- 不必要的精确定位。
- 未说明用途的通讯录等敏感数据。

### 14.2 删除与匿名化

用户删除账号时：

- 账号状态变为 deleted。
- 邮箱等登录标识可按策略清除或脱敏。
- 公开内容可保留但作者匿名化，需在用户协议中说明。
- 用户也应能删除自己的内容。

## 15. 审计日志

必须记录：

- 操作人。
- 操作对象。
- 操作前状态。
- 操作后状态。
- 操作原因。
- 操作时间。
- IP 和 user agent，可选。

审计日志不可由普通管理员删除。

## 16. 内容完整度机制

发布页可以展示完整度：

```text
当前完整度：72%

你已经填写：
- 礼物照片
- 送礼对象
- 场景
- 预算
- 为什么送

建议补充：
- 对方反应
- 踩坑提醒
- 准备时间
```

完整度不强制，但影响内容质量分和精选概率。

## 17. 验收标准

- 发布前显示隐私提醒，用户必须确认。
- 待审核内容不会在前台公开展示。
- 管理员能通过、驳回、隐藏内容并填写原因。
- 用户能举报故事和评论。
- 举报内容在后台可被处理。
- 外链和垃圾评论有基础限制。
- 匿名发布在前台不暴露作者身份。
- 审核操作写入审计日志。
- 上传文件限制类型和大小。
- 非管理员无法访问后台 API。
