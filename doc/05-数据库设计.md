# 数据库设计

## 1. 设计目标

数据库需要支撑 Gift 的核心闭环：

- 用户注册登录
- 发布礼物故事
- 图片管理
- 标签筛选
- 搜索排序
- 收藏
- 评论
- 很用心 / 感谢
- 我参考了这个
- 内容审核
- 举报处理
- 精选内容
- 数据埋点

MVP 推荐使用 PostgreSQL。本文以关系型设计为主，同时为后续搜索引擎、推荐系统和 AI 功能保留扩展字段。

## 2. 命名约定

- 表名使用复数 snake_case，例如 `gift_stories`。
- 主键统一使用 `id`，类型推荐 `uuid`。
- 时间字段统一使用 `created_at`、`updated_at`。
- 删除默认使用软删除字段 `deleted_at`，重要内容不直接物理删除。
- 状态字段使用枚举或受控字符串。
- 用户生成内容需要保留审核状态和可见性状态。

## 3. 核心实体关系

```text
users
  └── gift_stories
        ├── gift_story_images
        ├── gift_story_tags ── tags
        ├── comments
        ├── story_reactions
        ├── collection_items ── collections
        ├── gift_story_forks
        ├── reports
        └── moderation_logs
```

关键索引：

- unique `slug`
- index `(status, visibility, published_at desc)`
- index `(author_id, created_at desc)`
- index `(recipient_type, occasion, budget_range)`
- index `(is_failure_case, published_at desc)`
- GIN index `search_vector`
- GIN index on `title` using trigram，可选

业务约束：

- `approved + public` 的内容才能进入前台公开列表。
- `title` 不能是纯商品名，需在发布校验和审核中提示。
- `reason` 是核心必填字段。
- 如果已发布内容被作者修改，建议生成新的审核版本或重新进入 `pending`。

## 13. 搜索字段与索引策略

### 13.1 search_vector 来源

建议把以下字段合成 `search_vector`：

- title，权重 A
- summary，权重 B
- reason，权重 B
- reaction，权重 C
- tips，权重 C
- tag names，权重 A

### 13.2 排序辅助字段

`gift_stories` 中冗余：

- `quality_score`
- `save_count`
- `comment_count`
- `reaction_count`
- `fork_count`
- `published_at`

这些字段支持列表排序，不必每次聚合查询。

## 14. 数据状态机

### 14.1 礼物故事状态流转

```text
draft
  ↓ submit
pending
  ├── approve → approved + public
  ├── reject  → rejected + private
  └── hide    → hidden

approved
  ├── author edit → pending 或 draft_revision
  ├── admin hide  → hidden
  └── author delete → deleted

hidden
  ├── restore → approved
  └── delete  → deleted
```

### 14.2 举报状态流转

```text
pending
  ↓
reviewing
  ├── resolved
  └── rejected
```

## 15. 数据完整性规则

- 每个公开故事必须至少有一个封面图。
- 每个公开故事必须关联至少 3 个标签：送给谁、场景、预算。
- 收藏、点赞、感谢需要唯一约束防重复。
- 删除用户时，不直接删除公开内容，可匿名化作者。
- 审核日志不可删除。
- 举报记录不可物理删除。
- 管理员操作必须记录操作人和时间。

## 16. 数据迁移建议

MVP 初始化：

1. 创建用户、故事、标签、图片、收藏、评论、审核、举报表。
2. 初始化系统标签。
3. 创建默认管理员账号。
4. 导入 100-300 条冷启动内容。
5. 为冷启动内容批量关联标签。

后续迁移：

- P1 增加礼物计划、感谢机制、参考版本。
- P2 增加搜索索引同步表、推荐特征表。
- P3 增加 AI 解析结果、图片隐私检测结果。

## 17. 备份与恢复

- 生产数据库每日自动备份。
- 上线前执行迁移备份。
- 对象存储开启版本控制或定期备份。
- 备份恢复流程至少在 staging 环境演练一次。

## 18. 数据库验收标准

- 能通过迁移脚本从空库创建完整结构。
- 核心表具备必要索引。
- 发布、审核、搜索、收藏、评论、举报链路数据可追踪。
- 用户不能重复收藏同一礼物。
- 游客无法写入互动数据。
- 公开列表不会查出待审核、隐藏或删除内容。
- 删除用户后公开内容不会泄露用户私密信息。
