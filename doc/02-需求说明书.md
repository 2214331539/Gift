# 需求说明书

## 1. 需求范围

本文档定义 Gift Web 网站第一版和后续阶段的功能需求、非功能需求、业务规则和优先级。第一版以 MVP 为边界，优先建立“搜索 / 浏览真实案例 / 收藏 / 发布 / 审核”的核心闭环。

## 2. 用户角色

| 角色 | 说明 | 关键权限 |
| --- | --- | --- |
| 游客 | 未登录用户 | 浏览首页、列表、详情、搜索筛选、查看公开评论 |
| 注册用户 | 已登录普通用户 | 发布礼物、收藏、评论、点赞/很用心、举报、编辑自己的内容 |
| 内容作者 | 发布过礼物的用户 | 管理自己的礼物故事、查看互动反馈 |
| 审核员 | 管理后台人员 | 审核内容、处理举报、隐藏违规内容 |
| 管理员 | 平台运营/技术管理员 | 管理标签、用户、精选、后台权限、系统配置 |

## 4. 用户端需求

### 4.1 首页

需求编号：REQ-FE-HOME

首页目标：

1. 让急着找礼物的人快速进入搜索。
2. 让随便逛的人愿意继续浏览。
3. 让有经验的人愿意上传分享。

页面模块：

- 顶部导航
- 主搜索区
- 快捷入口
- 节日倒计时
- 今日精选礼物故事
- 热门场景入口
- 低预算高心意
- 最新分享
- 失败礼物避坑
- 分享礼物入口

业务规则：

- 搜索框支持自然语言输入，MVP 阶段可先做关键词分词和标签匹配。
- 快捷入口点击后进入礼物列表页，并自动带上筛选条件。
- 精选内容由后台手动配置，缺省时使用质量分和发布时间混合排序。
- 失败礼物内容在首页展示时应避免过度负面，用“避坑参考”语气呈现。

验收标准：

- 用户能从首页发起搜索并进入列表页。
- 首页至少展示 4 类内容入口：精选、热门场景、低预算、失败避坑。
- 未登录用户可以浏览首页。
- 登录用户可以从首页进入发布页。

### 4.2 礼物列表页

需求编号：REQ-FE-STORY-LIST

功能：

- 显示搜索结果和筛选结果。
- 支持关键词搜索。
- 支持多维筛选。
- 支持排序。
- 支持分页或无限加载。

筛选维度：

- 送给谁
- 场景
- 关系阶段
- 预算
- 准备时间
- 礼物类型
- 对方兴趣
- 情绪表达
- 是否失败避坑

排序方式：

- 最相关
- 最新
- 最多收藏
- 被参考最多
- 低预算优先
- 评分最高

业务规则：

- 默认排序为“最相关”，由标签匹配、内容质量、互动数据和发布时间综合计算。
- 预算筛选使用预算区间，不强制显示精确价格。
- 筛选条件应体现在 URL query 中，便于分享和返回。
- 列表只展示已审核通过、公开可见的内容。

验收标准：

- 用户选择筛选条件后，列表结果能刷新。
- 用户刷新页面后筛选条件不丢失。
- 无结果时展示可操作空状态：调整筛选、查看热门内容、发布自己的礼物。

### 4.3 礼物卡片

需求编号：REQ-FE-STORY-CARD

卡片字段：

- 封面图
- 标题
- 送给谁
- 场景
- 预算范围
- 准备时间
- 一句最打动人的描述
- 收藏数
- 被参考次数
- “很用心”数

展示原则：

- 不以商品名称、价格和购买按钮作为视觉重点。
- 标题应故事化，例如“给考研结束的女朋友做了一个重启生活礼盒”。
- 预算可以显示为区间，例如“100-300 元”或“未公开”。

验收标准：

- 卡片点击进入详情页。
- 图片缺失时有统一占位图，但发布时仍要求至少一张图。
- 卡片在移动端和桌面端信息不溢出。

### 4.4 礼物详情页

需求编号：REQ-FE-STORY-DETAIL

详情页目标：

1. 让用户判断是否适合自己的对象。
2. 让用户理解为什么这个礼物有效。
3. 让用户知道准备成本和风险。
4. 让用户可以收藏、评论、参考和感谢。

页面结构：

- 图片轮播
- 标题与核心信息
- 送礼背景
- 礼物内容
- 为什么选择这个礼物
- 准备过程
- 对方反应
- 预算与耗时
- 适合人群
- 不适合情况
- 踩坑提醒
- 互动操作
- 评论区
- 相似礼物推荐

核心字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 标题 | 是 | 故事化标题，不是纯商品名 |
| 图片 | 是 | 至少一张真实照片 |
| 送给谁 | 是 | 女朋友、男朋友、父母等 |
| 场景 | 是 | 生日、纪念日、毕业、道歉等 |
| 预算 | 是 | 预算区间，可隐藏具体价格 |
| 礼物类型 | 是 | DIY、数码、体验、实用、手写信等 |
| 为什么送 | 是 | 核心故事 |
| 对方反应 | 强引导 | 提高真实感 |
| 准备过程 | 建议 | 提高可参考性 |
| 踩坑提醒 | 建议 | 提高可信度 |
| 适合人群 | 建议 | 支撑搜索推荐 |
| 不适合情况 | 建议 | 避免盲目模仿 |

互动操作：

- 以后想送
- 很用心
- 评论
- 感谢作者
- 我参考了这个
- 举报

MVP 要求：

- “我参考了这个”在第一版可以先作为按钮和记录事件，P1 再做完整改造发布流程。

验收标准：

- 未登录用户可以浏览详情。
- 登录用户可以收藏、评论、点赞和举报。
- 作者可以编辑或删除自己的草稿/被驳回内容。
- 非作者不能编辑内容。

### 4.5 发布礼物

需求编号：REQ-FE-PUBLISH

发布目标：

发布流程必须轻量，用户不是来写论文或商品测评，而是记录“我送过这样一份礼物”。

推荐流程：

```text
第一步：选择送礼对象和场景
第二步：上传照片
第三步：填写礼物故事
第四步：选择标签和预算
第五步：设置隐私并发布
```

字段要求：

基础信息：

- 送给谁
- 送礼场景
- 关系阶段
- 预算区间
- 准备时间
- 礼物类型

故事信息：

- 标题
- 礼物内容
- 为什么想到送这个
- 准备过程
- TA 的反应
- 适合什么样的人
- 不适合什么情况
- 踩坑提醒

隐私设置：

- 匿名发布
- 隐藏具体价格
- 隐藏具体日期
- 图片隐私确认

业务规则：

- 发布前必须登录。
- 至少上传 1 张图片。
- 标题、送给谁、场景、预算、礼物类型、为什么送为必填。
- 发布后默认进入待审核状态。
- 审核通过后公开展示。
- 用户可以保存草稿。

验收标准：

- 用户可以完成完整发布流程。
- 必填字段缺失时不能提交，并清晰提示。
- 图片上传失败时可以重试。
- 发布成功后进入成功页，显示待审核提示。

### 4.6 收藏

需求编号：REQ-FE-COLLECTION

MVP 功能：

- 登录用户可以收藏礼物故事。
- 默认收藏夹名称为“以后想送”。
- 用户可以在个人主页查看收藏列表。
- 同一用户对同一故事不能重复收藏。

P1 功能：

- 多收藏夹
- 收藏备注
- 按对象或节日组织收藏
- 从收藏创建礼物计划

验收标准：

- 未登录用户点击收藏时引导登录。
- 登录用户收藏成功后按钮状态变化。
- 再次点击可取消收藏。

### 4.7 评论

需求编号：REQ-FE-COMMENT

MVP 功能：

- 登录用户可以发表评论。
- 支持一级回复。
- 作者和管理员可删除违规评论。
- 评论需要支持举报。

业务规则：

- 评论默认公开，但可以被审核系统隐藏。
- 评论内容长度限制为 2-1000 字。
- 评论不能包含明显垃圾链接。

验收标准：

- 评论发布后实时展示。
- 删除后前台不再展示正文，保留“评论已删除”状态可选。
- 举报后后台可见。

### 4.8 用户主页

需求编号：REQ-FE-PROFILE

个人主页目标不是炫耀，而是展示用户贡献。

页面内容：

- 用户昵称
- 头像
- 简介
- 发布过的礼物故事
- 收藏夹
- 被参考次数
- 收到的感谢
- 参与改造的礼物
- 失败避坑分享

指标展示原则：

不建议突出：

- 粉丝数
- 排名
- 热度榜第一

建议突出：

- 分享了多少个礼物故事
- 帮助了多少个送礼的人
- 收到多少条感谢
- 贡献多少条避坑提醒

## 5. 管理端需求

### 5.1 内容审核

需求编号：REQ-ADMIN-MODERATION

审核对象：

- 礼物故事
- 礼物图片
- 评论
- 用户举报内容

审核状态：

- draft 草稿
- pending 待审核
- approved 已通过
- rejected 已驳回
- hidden 已隐藏
- deleted 已删除

审核维度：

- 是否暴露隐私
- 是否有明显广告
- 是否盗图
- 是否攻击、歧视、低俗
- 是否包含违法或危险行为
- 是否标题党
- 是否重复搬运

验收标准：

- 审核员可以筛选待审核内容。
- 审核员可以通过、驳回、隐藏内容。
- 驳回必须填写原因。
- 所有审核操作写入审计日志。

### 5.2 举报处理

需求编号：REQ-ADMIN-REPORT

举报类型：

- 侵犯隐私
- 广告营销
- 虚假内容
- 恶意攻击
- 低俗内容
- 盗图
- 重复内容
- 其他

处理动作：

- 忽略
- 隐藏内容
- 要求作者修改
- 删除内容
- 限制用户
- 标记为重复举报

验收标准：

- 用户提交举报后后台可见。
- 管理员处理后状态变更。
- 处理结果记录处理人、时间和原因。

### 5.3 标签管理

需求编号：REQ-ADMIN-TAG

功能：

- 新增标签
- 编辑标签
- 合并重复标签
- 删除低质量标签
- 设置标签层级
- 维护热门标签
- 纠正错误标签

业务规则：

- 同义标签应合并到主标签，例如“异地恋”“异地”“远距离恋爱”。
- 系统标签和用户新增标签需要区分。
- 删除标签前要检查关联内容数量。

### 5.4 精选内容管理

需求编号：REQ-ADMIN-FEATURED

功能：

- 设置首页精选
- 设置低预算高心意
- 设置失败避坑推荐
- 设置节日专题内容
- 调整展示顺序
- 设置上线和下线时间

精选标准：

- 真实度高
- 有故事感
- 有参考价值
- 预算友好
- 非单纯贵重礼物
- 评论区反馈好
- 被多人参考
- 有后续更新

## 6. 非功能需求

### 6.1 性能

- 首页首屏服务端响应建议小于 500ms。
- 列表页搜索在常规筛选下建议小于 800ms。
- 图片使用压缩和 CDN。
- API P95 响应时间第一版目标小于 1000ms。
- 后台审核列表支持分页，避免一次性加载大量内容。

### 6.2 可用性

- 核心浏览功能对游客开放。
- 发布、收藏、评论等写操作要求登录。
- 发布流程支持草稿保存，避免用户中途丢失。
- 移动端必须可用，第一版以响应式 Web 为主。

### 6.3 安全

- 密码使用成熟认证方案，不自行明文存储。
- 所有写接口需要 CSRF 或同等防护。
- 用户输入统一做 XSS 防护。
- 图片上传限制类型、大小和数量。
- 后台接口必须校验权限。

### 6.4 隐私

- 支持匿名发布。
- 支持隐藏具体价格。
- 发布前必须展示隐私提醒。
- 删除账户时需要处理用户内容归属和匿名化策略。

### 6.5 可维护性

- 前后端统一 TypeScript 类型。
- API 有明确错误码。
- 数据库迁移纳入版本管理。
- 关键业务逻辑有单元测试或集成测试。
- 审核和权限相关逻辑必须有测试覆盖。

## 7. 业务规则摘要

| 规则 | 说明 |
| --- | --- |
| 内容公开前必须审核 | 防止广告和隐私泄露 |
| 至少一张图片 | 保证真实感和卡片展示质量 |
| 预算用区间 | 避免炫富和隐私压力 |
| 点赞命名为“很用心” | 强化用心文化，弱化热度攀比 |
| 收藏命名为“以后想送” | 更贴近用户意图 |
| 外链弱化 | 防止导购化 |
| 匿名发布可选 | 降低分享门槛 |
| 评论支持举报 | 保持社区安全 |
| 标签结构化 | 支撑搜索和推荐 |

## 8. 验收总则

一项功能完成需要同时满足：

1. 前端页面可用。
2. API 行为符合文档。
3. 数据正确落库。
4. 权限规则正确。
5. 错误状态有清晰提示。
6. 移动端布局可用。
7. 关键路径有测试或手工验收记录。
8. 不破坏“真实礼物故事社区”的定位。
