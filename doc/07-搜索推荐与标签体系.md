# 搜索推荐与标签体系

## 1. 设计目标

Gift 的搜索推荐目标不是推荐“最火”或“最贵”的礼物，而是帮助用户找到在具体关系、场景、预算和情绪下最适合参考的真实案例。

核心目标：

- 用户能用自然语言输入需求。
- 用户能用标签筛选缩小范围。
- 搜索结果优先匹配关系、场景、预算和情绪。
- 推荐结果保持多样性，避免昂贵礼物和热门礼物垄断曝光。
- 标签结构既能支撑 MVP，也能扩展到语义搜索和 AI 推荐。

## 2. 标签分类

### 2.1 标签类型总览

| 类型 | 示例 | 用途 |
| --- | --- | --- |
| recipient | 女朋友、男朋友、父母、朋友、同事 | 核心筛选 |
| relationship_stage | 暧昧期、刚在一起、1-3 年、异地、已婚 | 场景细化 |
| occasion | 生日、纪念日、毕业、道歉、考研结束 | 核心筛选 |
| budget | 50 元以内、100 元以内、300 元以内 | 核心筛选 |
| preparation_time | 当天、一周内、长期准备 | 判断可执行性 |
| interest | 摄影、游戏、手工、读书、运动 | 匹配对方兴趣 |
| emotion | 安慰、鼓励、陪伴、道歉、惊喜 | 匹配表达意图 |
| gift_type | DIY、实用、体验、数码、手写信 | 礼物形态 |
| quality | 真实照片、有对方反应、有避坑 | 内容质量 |
| topic | 低预算高心意、失败避坑、节日专题 | 运营专题 |

## 4. 标签规则

### 4.1 必填标签

发布礼物时至少需要：

- 送给谁
- 场景
- 预算
- 礼物类型

强引导标签：

- 情绪
- 兴趣
- 准备时间
- 关系阶段

### 4.2 同义词合并

示例：

```text
异地恋
异地
远距离恋爱
```

应合并为主标签：

```text
异地恋
```

数据库用 `canonical_id` 维护同义词关系。

### 4.3 标签来源

| 来源 | 说明 |
| --- | --- |
| user | 用户发布时选择 |
| system | 系统预置 |
| admin | 管理员修正 |
| ai | 后续 AI 自动建议 |

管理员修正优先级最高。

## 5. 搜索输入解析

用户可能输入：

```text
女朋友，生日，预算 500，喜欢拍照，最近工作压力大
```

解析结果：

```json
{
  "recipient": "女朋友",
  "occasion": "生日",
  "budgetMax": 500,
  "interest": ["摄影"],
  "emotion": ["安慰", "放松"],
  "keywords": ["工作压力大"]
}
```

MVP 可用规则解析：

- 命中标签词典。
- 命中预算表达。
- 剩余文本进入全文搜索。

后续可用 AI/LLM 解析：

- 提取关系、场景、预算、兴趣、情绪。
- 生成查询扩展词。
- 推荐筛选条件。

## 6. 搜索结果排序

### 6.1 排序目标

排序应优先保证：

```text
用户条件匹配 > 内容质量 > 互动数据 > 发布时间
```

不要简单按热度排序。

### 6.2 排序分组成

建议得分：

```text
final_score =
  match_score * 0.45 +
  quality_score * 0.25 +
  interaction_score * 0.15 +
  freshness_score * 0.10 +
  diversity_score * 0.05
```

### 6.3 match_score

匹配分来源：

- 关系匹配
- 场景匹配
- 预算匹配
- 兴趣匹配
- 情绪匹配
- 准备时间匹配
- 关键词全文匹配

建议权重：

| 因素 | 权重 |
| --- | --- |
| 关系 | 20 |
| 场景 | 20 |
| 预算 | 15 |
| 情绪 | 15 |
| 兴趣 | 10 |
| 礼物类型 | 10 |
| 准备时间 | 5 |
| 全文关键词 | 5 |

### 6.4 quality_score

内容质量分来源：

- 有真实照片
- 标题具体
- 场景清楚
- 预算明确
- 说明为什么送
- 有对方反应
- 有准备过程
- 有适合人群
- 有不适合情况或避坑提醒
- 非广告

示例：

```text
基础分 0
+15 有真实照片
+10 标题具体
+15 why/reason 字段长度足够
+15 有对方反应
+10 有准备过程
+10 有踩坑提醒
+10 有适合/不适合说明
+10 标签完整
+5 预算明确
-30 被多次举报未处理
-50 疑似广告
```

### 6.5 interaction_score

互动分来源：

- 收藏数
- 很用心数
- 评论质量
- 被参考次数
- 感谢数

注意：

- 不应让高互动内容永久霸榜。
- 使用对数处理，避免头部内容碾压。

示例：

```text
interaction_score =
  log(1 + save_count) * 3 +
  log(1 + reaction_count) * 2 +
  log(1 + fork_count) * 4 +
  log(1 + thanks_count) * 3
```

### 6.6 freshness_score

新内容需要适度曝光：

```text
freshness_score = exp(-days_since_published / 30)
```

### 6.7 diversity_score

用于打散：

- 不连续展示同一预算段。
- 不连续展示同一礼物类型。
- 不让全部结果都是数码或高价礼物。
- 低预算优质内容应有稳定曝光。

## 7. 首页推荐

首页推荐不是单一榜单，而是多个内容池混合：

```text
热门真实故事
最新发布
低预算高心意
失败避坑
节日相关
被多人参考
```

推荐比例建议：

| 内容池 | 比例 |
| --- | --- |
| 今日精选 | 25% |
| 最新发布 | 20% |
| 低预算高心意 | 20% |
| 节日相关 | 15% |
| 失败避坑 | 10% |
| 被多人参考 | 10% |

运营可通过后台手动覆盖。

## 8. 详情页相似推荐

详情页相似推荐按以下优先级：

1. 同场景 + 同关系 + 同预算。
2. 同场景 + 同情绪。
3. 同礼物类型 + 同兴趣。
4. 原礼物的改造版本。
5. 同主题的失败避坑案例。

结果中应避免：

- 推荐当前故事本身。
- 推荐隐藏或待审核内容。
- 推荐过多同质商品。

## 9. 失败避坑推荐

失败案例不是负面内容堆砌，而是帮助用户避免不适合的送礼方式。

失败案例标签：

- 对方不需要
- 关系阶段不合适
- 太贵有压力
- 太廉价显敷衍
- 使用场景不匹配
- 尺寸/型号错误
- 容易尴尬
- 隐私风险
- 准备时间不足

展示规则：

- 标题避免嘲讽。
- 强调“为什么不适合”和“替代方案”。
- 详情页可推荐相同场景下的成功案例。

## 10. 搜索反馈机制

P1 增加用户反馈：

- 这个结果有帮助。
- 这个结果不相关。
- 我想找更低预算。
- 我想找更有仪式感。
- 我想看失败避坑。

反馈用于：

- 调整标签。
- 优化排序。
- 发现缺失内容。
- 指导冷启动内容补充。

## 11. 冷启动搜索策略

上线前准备内容：

```text
至少 100 条
理想 300 条
```

内容覆盖：

- 送女朋友
- 送男朋友
- 送父母
- 异地恋
- 低预算
- 生日
- 纪念日
- 道歉
- 失败避坑
- DIY
- 实用礼物
- 体验型礼物

冷启动时标签必须由人工校正，保证第一批搜索体验。

## 12. 后续 AI 能力

AI 礼物助手不能直接替代真实案例。正确定位是：

- 理解用户需求。
- 召回真实案例。
- 总结可参考点。
- 建议筛选条件。
- 提醒隐私风险。

示例输出：

```text
根据你的描述，我优先帮你找了“女朋友 + 压力大 + 放松 + 300 元以内”的真实案例。
```

而不是直接输出空泛礼物清单。

## 13. 验收标准

- 用户输入包含关系、场景、预算时，搜索结果能明显匹配这些条件。
- 筛选条件可组合使用。
- 低预算内容不会被高价高互动内容完全挤出。
- 首页至少包含精选、最新、低预算、失败避坑四类内容。
- 详情页相似推荐与当前内容有明确关联。
- 标签后台可以新增、合并、禁用标签。
- 搜索无结果时提供可操作替代路径。
