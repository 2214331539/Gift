# 06. 数据库设计

数据库使用 PostgreSQL。主键统一使用 `uuid`，时间字段统一使用 `timestamptz`。所有业务表保留 `created_at`、`updated_at`，需要软删除的表保留 `deleted_at`。

## 1. 核心实体关系

```text
users 1----n gift_stories
users 1----n refresh_tokens
gift_stories n----n tags
gift_stories 1----n story_images
users 1----n collections 1----n collection_items
gift_stories 1----n comments
gift_stories 1----n reports
gift_stories 1----n gift_forks
gift_stories 1----n notifications
users 1----n gift_plans
users 1----n notifications
```

## 4. Refresh Token 表 refresh_tokens

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid pk | Token id |
| `user_id` | uuid fk | 用户 |
| `token_hash` | varchar | Token 哈希值（不存明文） |
| `expires_at` | timestamptz | 过期时间 |
| `revoked_at` | timestamptz | 吊销时间，null 表示有效 |
| `replaced_by` | uuid fk nullable | 轮换后的新 Token id |
| `user_agent` | text | 登录设备信息 |
| `ip_address` | varchar | 登录 IP |
| `created_at` | timestamptz | 创建时间 |

索引：

- `index(user_id, revoked_at)` — 查询用户有效 Token
- `index(token_hash)` — Token 查找
- `index(expires_at)` — 定期清理过期 Token

说明：

- Refresh Token 使用 Rotation 策略，每次刷新后旧 Token 标记 `revoked_at` 并记录 `replaced_by`。
- 如果检测到已吊销的 Token 被再次使用（可能被盗），立即吊销该用户所有 Token。
- 定时任务清理 `expires_at` 已过期的记录。

## 19. 数据一致性规则

- 故事被下架后，不从收藏夹删除，但普通用户访问时显示不可见。
- 用户匿名发布只影响前台展示，不影响后台追踪和风控。
- 删除标签前必须先迁移或清理 `story_tags`。
- 收藏数、评论数、参考数使用冗余字段，异步校准。
- 审核日志不可物理删除。
- Refresh Token 过期后由定时任务清理，不影响业务查询。
- 通知表数据量大时按 `created_at` 分区或定期归档超过 90 天的已读通知。
- `privacy` JSONB 字段变更不触发重新审核，仅影响前端展示逻辑。
- 用户被封禁时，立即吊销其所有 Refresh Token。
- 邮箱未验证的用户可以浏览和收藏，但不能发布内容和评论。
