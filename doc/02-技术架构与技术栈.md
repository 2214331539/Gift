# 02. 技术架构与技术栈

## 1. 架构原则

- Web MVP 优先：先完成可服务网站，不提前投入 App 和复杂微服务。
- 前后端严格分离：Next.js 只负责页面渲染和 SSR/SEO，所有业务逻辑、数据权限、审核在 NestJS 后端完成。Next.js 不承担 BFF 聚合职责，直接调用后端 API。
- 移动优先：送礼场景用户大概率在手机上搜索，所有页面以移动端为基准设计，桌面端做增强适配。
- 内容安全优先：UGC 图片和文本都必须走审核、举报和风控流程。
- 可扩展：搜索、推荐、AI、图片识别后续可以独立演进。
- 低运维复杂度：MVP 采用单体后端 + PostgreSQL + Redis + 对象存储即可。

## 2. 推荐技术栈

| 层级 | 技术选型 | 说明 |
| --- | --- | --- |
| 前端 | Next.js + React + TypeScript | 支持 SSR/SEO，适合内容社区 |

| 表单 | React Hook Form + Zod | 分步发布表单、校验和类型推导 |
| 国际化 | next-intl | MVP 只有中文，但从第一天用 i18n 包裹文案，降低后续多语言改造成本 |
| 后端 | NestJS + TypeScript | 模块化清晰，适合 API、权限、审核后台 |
| ORM | Prisma | 类型安全，迁移和开发效率高 |
| 数据库 | PostgreSQL | 适合关系数据、标签、审核、全文检索基础能力 |
| 缓存 | Redis | 限流、Token 黑名单、热门数据缓存、队列状态 |
| 搜索 | Meilisearch（MVP 即引入） | 中文分词开箱即用，PostgreSQL FTS 对中文需额外分词插件，直接用 Meilisearch 降低复杂度 |
| 文件存储 | S3 兼容对象存储 / MinIO | 存储礼物图片、分享卡片 |
| 图片处理 | Sharp + imgproxy | Sharp 做服务端裁剪/压缩/EXIF 剥离，imgproxy 做 CDN 层动态变换（WebP/AVIF、响应式尺寸） |
| 队列 | BullMQ + Redis | 图片处理、审核任务、通知任务 |
| 鉴权 | JWT（Access Token + Refresh Token） | Access Token 短期有效（15 分钟），Refresh Token 长期有效（7 天），存储于 HttpOnly Cookie |
| 部署 | Docker Compose 起步，后续 Kubernetes | MVP 简化部署，保留演进空间 |
| 网关 | Nginx / Caddy | HTTPS、反向代理、静态资源缓存 |
| 监控 | Prometheus + Grafana / 云监控 | 指标、日志、告警 |
| 异常追踪 | Sentry | 前端 JS 错误和后端 API 异常实时感知 |
| 日志 | Pino + Loki / 云日志 | 结构化日志，便于排查审核和风控问题 |

## 3. 系统架构

```text
Browser / Mobile Browser
  |
  v
Next.js Web (SSR + 静态页面，纯渲染层)
  |
  v
NestJS API (所有业务逻辑、鉴权、审核)
  |---- PostgreSQL
  |---- Redis (Token 黑名单、限流、队列)
  |---- Meilisearch (搜索索引)
  |---- Object Storage (图片)
  |---- imgproxy (图片动态变换)
  |---- BullMQ Worker (异步任务)
  |
  v
Admin Web (同一 Next.js 应用 /admin 路由)
```

前后端职责边界：

| 层 | 职责 | 不做 |
| --- | --- | --- |
| Next.js | SSR/SSG 渲染、SEO meta、页面路由、调用后端 API、前端状态管理 | 不写业务逻辑、不直接访问数据库、不做权限判断 |
| NestJS | 所有 CRUD、鉴权、权限校验、审核、搜索聚合、文件签名、通知、队列 | 不做页面渲染 |

MVP 把用户端和管理端前端放在同一个 Next.js 应用中，通过 `/admin` 路由和后端权限校验隔离。管理端 API 统一在 NestJS 的 AdminModule 中实现。

## 4. JWT 鉴权方案

### 4.1 Token 结构

| Token | 有效期 | 存储位置 | 用途 |
| --- | --- | --- | --- |
| Access Token | 15 分钟 | 前端内存（不存 localStorage） | API 请求认证 |
| Refresh Token | 7 天 | HttpOnly + Secure + SameSite=Lax Cookie | 静默刷新 Access Token |

### 4.2 流程

```text
登录成功
  -> 后端签发 Access Token + Refresh Token
  -> Refresh Token 写入 HttpOnly Cookie
  -> Access Token 返回 JSON body，前端存内存

API 请求
  -> 前端在 Authorization: Bearer <access_token> 中携带
  -> 后端校验签名和过期时间

Access Token 过期
  -> 前端收到 401
  -> 自动调用 POST /auth/refresh（Cookie 自动携带 Refresh Token）
  -> 后端校验 Refresh Token，签发新 Access Token
  -> 前端重试原请求

退出登录
  -> 后端将 Refresh Token 加入 Redis 黑名单（TTL = 剩余有效期）
  -> 清除 Cookie
```

### 4.3 安全措施

- Refresh Token 每次使用后轮换（Rotation），旧 Token 立即失效。
- Redis 维护 Token 黑名单，支持即时吊销。
- Access Token payload 只包含 `userId`、`role`、`exp`，不含敏感信息。
- CSRF 防护：API 使用 Authorization Header 认证（非 Cookie 自动携带），天然免疫 CSRF。Refresh 接口额外校验 Origin/Referer。
- 前端页面刷新时通过 Refresh Token 静默获取新 Access Token，无需用户重新登录。

## 8. SEO 与内容可访问性

- 礼物详情页和专题页使用 SSR/静态增量生成，便于搜索引擎收录。
- 标题格式：`给{对象}的{场景}礼物：{故事标题} - Gift`。
- 详情页结构化数据可按 Article 处理，不标记为 Product，避免导购站语义。
- 图片必须有 alt 文本，优先描述礼物内容和场景。

## 9. 图片处理流程

```text
用户选择图片
  -> 前端请求 POST /media/upload-token（获取 S3 预签名 URL）
  -> 前端直传 S3（限制：单文件 ≤ 10MB，MIME 仅 image/jpeg|png|webp|heic）
  -> 前端调用 POST /media/complete 通知后端
  -> 后端 Worker 异步处理：
     1. EXIF 剥离（Sharp）
     2. 生成缩略图（封面 400x300、列表 200x200）
     3. 转换 WebP 格式
     4. 可选：内容安全扫描（色情/暴力检测 API）
  -> imgproxy 提供动态 URL 变换（按设备 DPR 和宽度响应式出图）
```

上传凭证约束：

| 约束 | 值 |
| --- | --- |
| 单文件最大 | 10 MB |
| 允许 MIME | image/jpeg, image/png, image/webp, image/heic |
| 凭证有效期 | 10 分钟 |
| 单次最多上传 | 9 张 |
| 每日每用户上限 | 100 张 |
