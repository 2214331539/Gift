# 02. 技术架构与技术栈

## 1. 架构原则

- Web MVP 优先：先完成可服务网站，不提前投入 App 和复杂微服务。
- 前后端边界清晰：前端负责页面体验，后端负责业务规则、数据权限、审核和搜索聚合。
- 内容安全优先：UGC 图片和文本都必须走审核、举报和风控流程。
- 可扩展：搜索、推荐、AI、图片识别后续可以独立演进。
- 低运维复杂度：MVP 采用单体后端 + PostgreSQL + Redis + 对象存储即可。

## 2. 推荐技术栈

| 层级 | 技术选型 | 说明 |
| --- | --- | --- |
| 前端 | Next.js + React + TypeScript | 支持 SSR/SEO，适合内容社区 |
| UI | Tailwind CSS + shadcn/ui + lucide-react | 快速构建一致的界面和图标体系 |
| 状态 | TanStack Query + Zustand | 服务端状态和局部交互状态分离 |
| 表单 | React Hook Form + Zod | 分步发布表单、校验和类型推导 |
| 后端 | NestJS + TypeScript | 模块化清晰，适合 API、权限、审核后台 |
| ORM | Prisma | 类型安全，迁移和开发效率高 |
| 数据库 | PostgreSQL | 适合关系数据、标签、审核、全文检索基础能力 |
| 缓存 | Redis | 会话、限流、热门数据缓存、队列状态 |
| 搜索 | PostgreSQL FTS 起步，后续接 Meilisearch / OpenSearch | MVP 先低成本实现，后续提升相关性 |
| 文件存储 | S3 兼容对象存储 / MinIO | 存储礼物图片、分享卡片 |
| 队列 | BullMQ + Redis | 图片处理、审核任务、通知任务 |
| 鉴权 | HttpOnly Cookie + JWT / Session | Web 端安全性优先 |
| 部署 | Docker Compose 起步，后续 Kubernetes | MVP 简化部署，保留演进空间 |
| 网关 | Nginx / Caddy | HTTPS、反向代理、静态资源缓存 |
| 监控 | Prometheus + Grafana / 云监控 | 指标、日志、告警 |
| 日志 | Pino + Loki / 云日志 | 结构化日志，便于排查审核和风控问题 |

## 3. 系统架构

```text
Browser
  |
  v
Next.js Web
  |
  v
NestJS API
  |---- PostgreSQL
  |---- Redis
  |---- Object Storage
  |---- Search Engine
  |---- Queue Worker
  |
  v
Admin Web
```

MVP 可把用户端和管理端放在同一个 Next.js 应用中，通过 `/admin` 路由和权限控制隔离。后续后台复杂后再拆独立应用。

## 4. 后端模块划分

| 模块 | 责任 |
| --- | --- |
| AuthModule | 注册、登录、退出、会话刷新、密码重置 |
| UserModule | 用户资料、个人主页、贡献统计 |
| StoryModule | 礼物故事 CRUD、状态流转、详情聚合 |
| MediaModule | 图片上传、裁剪、封面、对象存储签名 |
| TagModule | 标签分类、标签合并、热门标签 |
| SearchModule | 搜索、筛选、排序、搜索日志 |
| CollectionModule | 收藏夹、收藏项 |
| CommentModule | 评论、回复、删除、举报入口 |
| ReactionModule | “很用心”、感谢、参考行为 |
| ForkModule | “我也参考了这个”版本关系 |
| ReportModule | 用户举报 |
| ModerationModule | 内容审核、敏感词、广告识别、审核日志 |
| AdminModule | 管理后台接口 |
| NotificationModule | 站内通知、审核结果、被参考提醒 |
| PlanModule | 礼物计划、节日提醒 |

## 6. 环境划分

| 环境 | 用途 | 数据策略 |
| --- | --- | --- |
| local | 本地开发 | 本地 PostgreSQL / Docker Compose |
| dev | 联调环境 | 可重置测试数据 |
| staging | 预发布 | 接近生产配置，使用脱敏数据 |
| production | 线上 | 严格备份、权限和告警 |

## 7. SEO 与内容可访问性

- 礼物详情页和专题页使用 SSR/静态增量生成，便于搜索引擎收录。
- 标题格式：`给{对象}的{场景}礼物：{故事标题} - Gift`。
- 详情页结构化数据可按 Article 处理，不标记为 Product，避免导购站语义。
- 图片必须有 alt 文本，优先描述礼物内容和场景。

## 8. 技术演进路径

| 阶段 | 方案 |
| --- | --- |
| MVP | PostgreSQL FTS + 规则推荐 + 单体 NestJS |
| V1.5 | 增加 Meilisearch，优化搜索召回和拼写容错 |
| V2.0 | 礼物版本树、推荐特征表、异步统计 |
| V3.0 | AI 礼物助手、向量检索、图片隐私识别 |
