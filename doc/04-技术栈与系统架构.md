# 技术栈与系统架构

## 1. 架构目标

Gift 第一版应采用简单、可维护、可快速部署的架构。不要在 MVP 阶段引入过度复杂的微服务、推荐系统和 AI 管线。

架构目标：

- 支持真实用户注册、发布、浏览、搜索、收藏、评论和审核。
- 支持图片上传与 CDN 访问。
- 支持结构化标签检索和基础全文搜索。
- 支持后台审核和举报处理。
- 支持后续扩展到语义搜索、AI 推荐和礼物版本树。
- 保持单人或小团队可维护。

## 2. 推荐技术栈

### 2.1 MVP 推荐方案

| 层级 | 推荐技术 | 说明 |
| --- | --- | --- |
| 前端框架 | Next.js App Router + React + TypeScript | 同时支持服务端渲染、API 路由和 SEO |
| 样式 | Tailwind CSS | 快速构建设计一致的响应式界面 |
| 组件 | shadcn/ui 或自建基础组件 | 控制 UI 质量，避免过度设计 |
| 表单 | React Hook Form + Zod | 发布流程字段多，需要强校验 |
| 服务端 | Next.js Route Handlers / Server Actions | MVP 保持单体全栈工程 |
| ORM | Prisma | 类型安全、迁移清晰 |
| 数据库 | PostgreSQL | 结构化标签、全文检索和关系查询都适合 |
| 鉴权 | Auth.js / NextAuth 或同级成熟方案 | 不自研认证 |
| 图片存储 | S3 兼容对象存储，例如 Cloudflare R2、阿里云 OSS、AWS S3 | 存储用户上传图片 |
| 图片处理 | sharp 或对象存储图片处理能力 | 缩略图、压缩、格式转换 |
| 缓存 | Redis，可 MVP 后引入 | 热门标签、首页精选、限流 |
| 搜索 | PostgreSQL tsvector + trigram，后续 Meilisearch/OpenSearch | MVP 先用数据库能力 |
| 部署 | Vercel / Render / Railway / 自建服务器 + Docker | 按预算和访问地区选择 |
| 日志监控 | Sentry + 平台日志 | 捕获前后端错误 |
| 数据统计 | Plausible / Umami / 自建事件表 | 跟踪北极星指标 |

### 2.2 为什么推荐单体全栈

Gift 第一版的复杂度主要来自产品规则，而不是高并发技术瓶颈。单体全栈更适合：

- 快速验证 MVP。
- 前后端类型共用。
- 减少部署和联调成本。
- 小团队更容易维护。
- 后续可以按模块拆分。

### 2.3 可替代方案

如果团队更熟悉其他栈，也可以替换：

| 场景 | 可替代方案 |
| --- | --- |
| 后端希望独立 | NestJS / Fastify + Next.js 前端 |
| 希望快速后端托管 | Supabase + Next.js |
| 国内云部署 | Next.js standalone + Docker + PostgreSQL + OSS |
| 重后台系统 | React Admin / Ant Design Pro + 独立 API |

## 5. 请求链路

### 5.1 浏览礼物列表

```text
浏览器请求 /stories?occasion=birthday&budget=100-300
    ↓
Next.js 页面读取 query
    ↓
调用服务端查询函数
    ↓
PostgreSQL 执行筛选、全文搜索、排序
    ↓
返回卡片数据
    ↓
服务端渲染首屏 + 客户端增强筛选交互
```

### 5.2 发布礼物

```text
用户填写发布表单
    ↓
前端 Zod 校验
    ↓
图片上传到对象存储，返回 mediaId
    ↓
提交故事数据
    ↓
服务端鉴权和字段校验
    ↓
写入 gift_stories，状态 pending
    ↓
写入 story_images 和 story_tags
    ↓
返回发布成功
    ↓
后台出现待审核记录
```

### 5.3 审核通过

```text
审核员打开后台
    ↓
查看 pending 内容
    ↓
通过审核
    ↓
更新 moderation_status = approved
    ↓
更新 visibility = public
    ↓
写入 moderation_logs
    ↓
通知作者
    ↓
内容进入前台检索和展示
```

## 7. 数据流设计

### 7.1 服务端优先

适合服务端渲染的数据：

- 首页精选内容
- 礼物列表首屏
- 礼物详情
- 标签列表
- 节日专区

适合客户端请求的数据：

- 收藏状态
- 评论分页加载
- 点赞 / 很用心状态
- 发布表单草稿自动保存
- 后台审核操作后的局部刷新

### 7.2 类型共享

建议把 API 请求和响应结构定义为 Zod schema：

```text
Zod Schema
    ↓
服务端校验
    ↓
TypeScript 类型推导
    ↓
前端表单和 API 调用复用
```

## 8. 鉴权与权限

### 8.1 登录方式

MVP 推荐：

- 邮箱 + 密码
- 邮箱验证码
- 后续增加第三方登录

### 8.2 角色

```text
guest
user
moderator
admin
```

### 8.3 权限原则

- 游客只能读公开内容。
- 登录用户可以写自己的内容和互动。
- 作者只能编辑自己的草稿、待审核或被驳回内容。
- 已公开内容编辑后应重新进入审核或产生修订版本。
- 审核员不能随意修改用户正文，只能处理状态和原因。
- 管理员可以管理标签、精选、用户和后台权限。

## 9. 图片上传架构

### 9.1 上传流程

```text
前端请求上传凭证
    ↓
服务端校验登录和文件元信息
    ↓
生成对象存储预签名 URL
    ↓
前端直传对象存储
    ↓
前端通知服务端确认上传
    ↓
服务端创建 media_assets 记录
```

### 9.2 图片规格

建议生成：

- 原图：限制最大边长和文件大小。
- 封面图：用于卡片，建议 4:3 或 1:1 裁剪。
- 缩略图：用于后台列表和小卡片。

### 9.3 限制

MVP 建议：

- 单张图片最大 8MB。
- 每篇故事最多 9 张图片。
- 支持 JPG、PNG、WebP。
- 禁止 SVG 上传，降低 XSS 风险。

## 10. 搜索架构

### 10.1 MVP 搜索

使用 PostgreSQL：

- `tsvector` 做标题、摘要、正文全文搜索。
- `pg_trgm` 做模糊匹配。
- 标签关联表做结构化筛选。
- 排序使用匹配分 + 质量分 + 互动分 + 时间衰减。

### 10.2 后续搜索

当内容量超过数万条，或需要更好的中文分词和语义搜索时，可引入：

- Meilisearch
- OpenSearch
- Elasticsearch
- 向量数据库，配合 embedding

## 11. 异步任务

MVP 可以先同步处理，以下任务后续建议异步化：

- 图片压缩和缩略图生成
- 内容质量分计算
- 搜索索引刷新
- 通知发送
- 数据统计聚合
- 敏感词扫描
- 图片隐私识别

推荐队列：

- BullMQ + Redis
- 云厂商队列服务
- 简单定时任务，MVP 可先用

## 12. 部署架构

### 12.1 最简生产部署

```text
Next.js 应用
PostgreSQL 数据库
对象存储
域名 + HTTPS
错误监控
备份任务
```

### 12.2 稳定生产部署

```text
CDN
    ↓
Next.js Web 服务
    ↓
PostgreSQL 主库
    ↓
对象存储
    ↓
Redis
    ↓
后台 Worker
```

### 12.3 环境划分

- local：本地开发
- staging：预发布验证
- production：正式环境

每个环境应独立配置数据库、对象存储 bucket、回调地址和密钥。

## 13. 可扩展路径

### 13.1 从单体到模块化

MVP：

```text
Next.js 单体全栈 + PostgreSQL
```

增长期：

```text
Next.js + 独立 Worker + Redis + 搜索引擎
```

规模化：

```text
Web 前端
API 服务
搜索服务
推荐服务
审核服务
媒体服务
```

### 13.2 AI 能力接入点

后续 AI 不应替代真实案例，而应帮助用户更快找到案例：

- 自然语言需求解析
- 标签自动推荐
- 相似案例召回
- 发布内容润色建议
- 隐私风险提示

AI 输出必须优先指向真实案例，避免变成空泛建议。

## 14. 架构验收标准

- 本地可以一条命令启动 Web 服务和数据库。
- 数据库迁移可重复执行。
- 图片上传在开发和生产环境都有明确配置。
- 用户端和管理端权限隔离。
- 搜索、发布、审核、收藏、评论链路完整。
- API 错误有统一格式。
- 关键操作有审计日志。
- 生产环境具备备份、日志和错误监控。
